OBO 调度器

队列永远只有
一个任务

解决消息积压导致的卡顿，同时让每个任务成为独立消息，让性能监控能精准定位"就是这个 SDK 慢了 51ms"。

消息积压消除 三方 SDK 拦截 来源级诊断 ASM 字节码改写

问题背景

问题一：消息积压 → 卡顿
大量异步任务同时提交到主线程队列，队列瞬间堆满。Android 屏障消息无法打断已积压的任务，渲染帧被阻塞在队列后方，掉帧不可避免。
问题二：卡顿了但不知道谁慢
几十个任务挤在一起执行，性能监控只能看到"这一批加起来耗时 200ms"，无法区分具体是哪个任务、哪个 SDK 造成的。

解决方案

OBO（One By One）的核心原则：队列里永远最多只有一个任务。当前任务执行完后才提交下一个。

效果

VSYNC 到达时队列几乎是空的，屏障机制正常生效，渲染帧顺畅执行。同样工作量（30 个列表项 × 10 个任务 × 3ms），原生并发会卡死，OBO 逐个执行保持丝滑滚动。

取舍

OBO 让整批任务的总完成时间变长，但用户不会感知后台任务"2 秒 vs 1 秒完成"，却一定会感知"滑动时卡了 1 秒"。流畅优先于速度，这是正确的取舍。

内置对比界面

压测页面带实时帧率指示器 + 可调参数，一键切换 OBO / 原生模式，效果差异肉眼可见，可直观验证。

Handler 拦截

OBO 只能管住项目自身代码。三方 SDK（IM / 支付 / 归因等）直接调 Handler.post/send 往主线程塞消息，绕过 OBO 照样造成积压。

解决方案：编译期 ASM 字节码改写，将三方 SDK 的 Handler 调用自动收束到 OBO，无需 SDK 配合。

编译期： INVOKEVIRTUAL Handler.post(Runnable) → LDC "com.thirdparty.SomeClass" ← 编译期注入调用方类名 → INVOKESTATIC OBOCompat.post(Handler, Runnable, String) 运行时： 立即消息 → OBOScheduler.post(tag) { r.run() } 延时消息 → handler.postDelayed({ OBOScheduler.post(tag) { r.run() } }, delay) 定时消息 → handler.postAtTime({ OBOScheduler.post(tag) { r.run() } }, time) 队首消息 → OBOScheduler.postFirst(tag) { r.run() }

核心原则：定时交给系统 MessageQueue，执行交给 OBO。

拦截的 13 个 Handler 方法

类别方法
立即post, sendMessage, sendEmptyMessage
延时postDelayed(×2), sendMessageDelayed, sendEmptyMessageDelayed
定时postAtTime(×2), sendMessageAtTime, sendEmptyMessageAtTime
队首postAtFrontOfQueue, sendMessageAtFrontOfQueue

拦截报告

独立 Gradle task，只读扫描 runtime classpath JAR，支持增量缓存。报告路径：

androidApp/build/reports/obo-handler/{variant}-intercept.txt
格式：类名 | 方法() → Handler.xxx ×次数

前缀原因

运行时诊断

开启 AppConfig.enableOBODiagnostics = true 后，追踪每个任务的执行耗时和来源。

[OBO:SLOW] 慢任务

单个任务执行超过半帧（阈值跟设备刷新率挂钩）：

[OBO:SLOW] com.thirdparty.im.sdk.IMClient 12.3ms (threshold=8.3ms)
刷新率阈值
60Hz8.3ms
90Hz5.6ms
120Hz4.2ms

[OBO:FLOOD] 高频灌入

1s 窗口内某来源累计执行超过 100ms（帧预算的 10%），冷却期 5s：

[OBO:FLOOD] com.thirdparty.im.push.f 185.0ms/1000ms (threshold=100.0ms, count=92, cooldown=5s)

全局开关

开关作用
AppConfig.useOBOScheduler = false关闭 OBO 调度，OBOCompat 透传原始 Handler 调用
AppConfig.enableOBODiagnostics = true开启来源级诊断日志（每任务 2× nanoTime + HashMap 操作额外开销）
安全保障

com.example.archshowcase.core.scheduler.* 包内所有类（含 OBOCompat）被排除改写，避免递归。非主线程 Handler 直接透传，延时/定时消息的时序语义完整保留。
