去 Material3 智能跳过 延迟读取 计算缓存

为什么去 Material3

Google 官方 Material3 组件在易用性上做了很多封装，但这些封装隐含了额外的重组开销：

隐含状态订阅
官方组件内部订阅了多个 CompositionLocal（主题、颜色、形状），任何主题变化都会触发所有组件重组，即使视觉没有变化。
过度封装
Button / Text / Card 等组件包裹了多层 Composable，每层都增加重组检查的开销。轻量自研组件直接去掉不必要的层级。
稳定性声明困难
官方组件参数中有大量未标 @Stable 的类型，导致编译器无法推断"参数没变就跳过重组"，只能保守地每次都执行。
自研可控
自研组件完全掌控稳定性声明，精确标注哪些参数可以触发跳过优化，重组范围可以精确到字段级别。

智能跳过（Smart Skipping）

Compose 编译器会在每个 @Composable 函数入口生成跳过检查：如果所有参数与上次相同，直接跳过整个函数体，不执行任何代码。

前提：参数必须是 Stable 类型

只有当所有参数都被标记为 @Stable 或编译器能推断为稳定，跳过优化才会生效。不稳定类型会让编译器保守地每次都执行。

稳定性声明方式

// 方式一：@Stable 标注（适用于接口 / 抽象类）
@Stable
interface ImageItem {
    val id: String
    val url: String
}

// 方式二：@Immutable 标注（适用于不可变数据）
@Immutable
data class ImageCard(val id: String, val url: String)

// 方式三：Stability Config（全局声明第三方类型为稳定）
// compose_compiler_config.conf：
// com.example.SomeThirdPartyClass

延迟读取（Deferred Reading）

高频变化的值（动画进度、滚动偏移）如果在父组件读取，会导致父组件频繁重组，连带子树全部重组。延迟读取把这类值的读取推迟到最后一刻，只在绘制阶段读取，跳过重组阶段。

// ❌ 错误：在父组件读取动画值，父组件每帧都重组
@Composable
fun ParentCard(animProgress: Float) {
    Box(Modifier.alpha(animProgress)) { /* 子树每帧都重组 */ }
}

// ✅ 正确：传入 lambda，延迟到绘制阶段读取
@Composable
fun ParentCard(animProgress: () -> Float) {
    Box(Modifier.graphicsLayer { alpha = animProgress() }) {
        /* 子树不重组，只有 graphicsLayer 更新 */
    }
}
适用场景

动画进度、滚动偏移、实时数据（帧率计数器等）。这类值每帧都变，不适合触发重组，应通过 graphicsLayer / drawWithContent 在绘制阶段消费。

计算缓存（remember + derivedStateOf）

排序、过滤、格式化等计算如果每次重组都执行，在长列表场景下会造成明显卡顿。用 remember 和 derivedStateOf 缓存结果，数据不变不重复计算。

// ❌ 错误：每次重组都重新排序
val sorted = items.sortedBy { it.timestamp }

// ✅ 正确：只有 items 变化时才重新排序
val sorted = remember(items) {
    items.sortedBy { it.timestamp }
}

// ✅ 更好：derivedStateOf 细粒度订阅
val unreadCount by remember {
    derivedStateOf { messages.count { !it.isRead } }
}
// unreadCount 只在 messages 中未读数量变化时才更新
场景推荐方式
排序 / 过滤（输入是整个列表）remember(key) { ... }
派生计数 / 判断（输入是 State）derivedStateOf { ... }
格式化字符串（输入是数据字段）remember(field) { format(field) }

列表滚动优化

长列表滚动时，视觉效果实现方式对重组频率影响巨大。

// ❌ 错误：clip + shape 触发 Layout 重算
Box(
    Modifier
        .clip(RoundedCornerShape(12.dp))
        .background(color)
) { ... }

// ✅ 正确：graphicsLayer 绕过重组，直接在绘制层处理圆角
Box(
    Modifier.graphicsLayer {
        shape = RoundedCornerShape(12.dp)
        clip = true
    }
) { ... }
原理

graphicsLayer 的修改在 RenderThread 上应用，不触发 Compose 的 measure / layout / composition 阶段。滚动时圆角、阴影等视觉效果可以完全绕开重组。

如何验证重组效果

Android Studio 提供实时工具，可以直观看到哪些组件在重组。

01

打开 Running Devices + Layout Inspector

Android Studio → View → Tool Windows → Running Devices，连接设备后打开 Layout Inspector。

02

开启重组计数

Layout Inspector 顶部 → 打开 "Show Recomposition Counts"，正在重组的组件会高亮显示，计数实时更新。

03

滚动列表观察

滚动列表时，只有屏幕内新出现的 item 会有重组计数，屏幕外和数据未变的 item 计数不增加，说明跳过优化生效。

这不是理论推导

可以随时打开工具验证。如果发现某个组件重组计数异常高，说明它的参数稳定性有问题，需要检查参数类型的 @Stable 标注。
