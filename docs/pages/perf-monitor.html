JANK — 卡顿诊断

按严重程度着色：SLIGHT（蓝）/ MODERATE（黄）/ SEVERE+FROZEN（红）。

[PERF:JANK] SEVERE 108ms (11 dropped) route=ImageDemo #912 (+19)
  ├─ Choreographer.doFrame 107ms
  ├─ gc: 2x blocking 15ms, 3x concurrent 45ms
  ├─ cpu: 100ms/108ms (92%)
  └─ intents: Push(route=ImageDemo), LoadInitial
[PERF:JANK] #912 Choreographer.doFrame拆解: delay 5ms | anim(recompose→apply) 30ms | layout 40ms | draw(m&l→canvas) 25ms | gpu 5ms (total 108ms)

FrameMetrics doFrame 6 阶段拆解

字段含义
delay帧排队等待（上一帧未完成 / 主线程忙）
anim(recompose→apply)performRecompose（计算 UI 树差异）+ applyChanges（提交 diff）
layoutView measure + layout（Compose 中通常 ≈0ms）
draw(m&l→canvas)measureAndLayout（Compose 测量+布局）+ LayoutNode.draw（Canvas 录制）
sync主线程 nativeSync（display list 提交到 RenderThread，主线程阻塞等待）
gpuRenderThread DrawFrame（API 31+）

GC 行

类型含义对卡顿的影响
blocking阻塞式 GC，暂停所有线程高：时间直接计入卡顿
concurrent并发 GC，后台线程执行低：争抢 CPU 但不暂停主线程

CPU 行

CPU%诊断
>80%代码本身重（渲染复杂、计算密集）
<80%主线程被抢占（锁等待 / IO 阻塞 / GC 停顿 / CPU 竞争）

GC_PRESSURE — GC 压力检测

5s 滑动窗口监控 blocking GC。分值 = (blockingGcTimeMs/sec) × (1 - freePercent/100)。10s 冷却防刷屏。

[PERF:GC_PRESSURE] SEVERE pressure=72.3 | 5 GC 180ms / 5s | heap 128/256MB (50% free)
  PSS: Java 48MB  Native 32MB  Code 12MB  Stack 2MB  Graphics 8MB
  ChatRoomStore
  ├─ messageWindow.messages: 1200
  │  └─ body: 1200
  └─ history: 850
  ConversationListStore
  └─ conversations: 500
级别阈值含义
WARN≥20轻度压力
SEVERE≥60影响帧率
CRITICAL≥150可能触发 LMK

Store 集合大小快照由 KSP @MemoryTrackable 自动生成，递归统计 Collection/Map .size，无需手写。

PERF 日志 ↔ Perfetto Trace 映射

结合 btrace 抓取的 Perfetto trace，可将 PERF 日志的聚合数据映射到方法级 slice。

Compose 帧渲染模型关键

Compose 不走 View 系统的 onMeasure/onLayout，layout 字段通常为 0。measureAndLayout 有两个出现位置（ANIMATION callback 内 + TRAVERSAL callback 内），分析 trace 时必须通过父链区分。

示例：帧拆解日志 vs Trace

日志: delay 21ms | anim(recompose→apply) 69ms | draw(m&l→canvas) 18ms | gpu 1ms

Trace 对应:
doFrame 88ms
├─ CallbackRecord.run (ANIMATION)
│  └─ runRecomposeAndApplyChanges 69ms
│     ├─ performRecompose 55ms       ← 重组：计算 UI 树差异
│     └─ applyChanges 11ms          ← 提交 diff 到 SlotTable
└─ TraversalRunnable (TRAVERSAL) ≈ draw 18ms
   └─ performDraw
      ├─ measureAndLayout 13ms       ← Compose 测量+布局
      └─ LayoutNode.draw ~5ms*       ← Canvas 录制
                                       *btrace 坍缩为 0ms
+ RenderThread Running 1.4ms         ≈ gpu 1ms
btrace draw 耗时修正

btrace 会将 TraversalRunnable 内的嵌套 slice 坍缩为相同 dur，导致 LayoutNode.draw 显示 0ms。在 Perfetto 中对比 slice 右边缘与 thread_state Running 右边缘的差，即为被吞掉的耗时。

排查流程

1. 看 JANK 日志确定卡顿类型 ├─ Choreographer.doFrame 慢 → 渲染问题 → 看 FrameMetrics 分解 │ ├─ anim 大 → recompose 重 → 检查重组范围 │ ├─ draw 大 → measureAndLayout 或 Canvas 录制重 → 简化布局 │ └─ delay 大 → 流水线阻塞 → 上一帧太慢导致排队 ├─ 其他 Message 慢 → 主线程有耗时任务 → 移到后台线程 ├─ gc blocking → GC 停顿 → 减少对象分配 ├─ cpu% 低 → 主线程被阻塞 → btrace 抓堆栈定位锁/IO └─ 无 msg + cpu% 低 → 线程调度延迟 / GC 并发期 2. btrace 抓方法级耗时，Perfetto 中 Ctrl+F 搜 "PERF:JANK" 直接跳到卡顿帧 3. 修复后对比 JANK 日志验证
