Removed Google's official Material3 component library and built a lightweight custom component system. Three strategies work together -- smart skipping, deferred reading, and computation caching -- ensuring only components whose data actually changed get recomposed while all others are skipped.

Why Drop Material3

Google's official Material3 components provide convenient abstractions, but these abstractions introduce hidden recomposition overhead:

Implicit State Subscriptions
Official components internally subscribe to multiple CompositionLocals (theme, color, shape). Any theme change triggers recomposition of all components, even when there is no visual change.
Over-Abstraction
Components like Button / Text / Card wrap multiple layers of Composables, each adding recomposition-check overhead. Lightweight custom components eliminate unnecessary layers.
Stability Declaration Difficulty
Many parameter types in official components lack @Stable annotations, preventing the compiler from inferring "skip recomposition if parameters are unchanged" and forcing conservative re-execution every time.
Full Control with Custom Components
Custom components give full control over stability declarations, allowing precise annotation of which parameters can trigger skip optimization, with recomposition scope narrowed down to the field level.

Smart Skipping

The Compose compiler inserts skip checks at the entry of every @Composable function: if all parameters are the same as last time, the entire function body is skipped without executing any code.

Prerequisite: Parameters Must Be Stable Types

Skip optimization only takes effect when all parameters are annotated with @Stable or the compiler can infer them as stable. Unstable types force the compiler to conservatively re-execute every time.

Stability Declaration Approaches

// Approach 1: @Stable annotation (for interfaces / abstract classes)
@Stable
interface ImageItem {
    val id: String
    val url: String
}

// Approach 2: @Immutable annotation (for immutable data)
@Immutable
data class ImageCard(val id: String, val url: String)

// Approach 3: Stability Config (globally declare third-party types as stable)
// compose_compiler_config.conf:
// com.example.SomeThirdPartyClass

Deferred Reading

Frequently changing values (animation progress, scroll offset) read in a parent component cause the parent to recompose often, dragging the entire subtree along. Deferred reading postpones reading these values to the last possible moment -- consuming them only during the draw phase and skipping the recomposition phase.

// ❌ Wrong: reading animation value in parent, parent recomposes every frame
@Composable
fun ParentCard(animProgress: Float) {
    Box(Modifier.alpha(animProgress)) { /* subtree recomposes every frame */ }
}

// ✅ Correct: pass a lambda, defer reading to the draw phase
@Composable
fun ParentCard(animProgress: () -> Float) {
    Box(Modifier.graphicsLayer { alpha = animProgress() }) {
        /* subtree does not recompose, only graphicsLayer updates */
    }
}
Applicable Scenarios

Animation progress, scroll offset, real-time data (frame rate counters, etc.). These values change every frame and should not trigger recomposition -- they should be consumed during the draw phase via graphicsLayer / drawWithContent.

Computation Caching (remember + derivedStateOf)

Computations like sorting, filtering, and formatting that run on every recomposition can cause noticeable jank in long-list scenarios. Use remember and derivedStateOf to cache results so they are not recomputed when data has not changed.

// ❌ Wrong: re-sorts on every recomposition
val sorted = items.sortedBy { it.timestamp }

// ✅ Correct: re-sorts only when items change
val sorted = remember(items) {
    items.sortedBy { it.timestamp }
}

// ✅ Better: derivedStateOf for fine-grained subscription
val unreadCount by remember {
    derivedStateOf { messages.count { !it.isRead } }
}
// unreadCount only updates when the unread count in messages changes
ScenarioRecommended Approach
Sorting / Filtering (input is the entire list)remember(key) { ... }
Derived count / boolean (input is State)derivedStateOf { ... }
String formatting (input is a data field)remember(field) { format(field) }

List Scroll Optimization

During long list scrolling, how visual effects are implemented has a significant impact on recomposition frequency.

// ❌ Wrong: clip + shape triggers Layout recalculation
Box(
    Modifier
        .clip(RoundedCornerShape(12.dp))
        .background(color)
) { ... }

// ✅ Correct: graphicsLayer bypasses recomposition, handles rounded corners at the draw layer
Box(
    Modifier.graphicsLayer {
        shape = RoundedCornerShape(12.dp)
        clip = true
    }
) { ... }
How It Works

graphicsLayer modifications are applied on the RenderThread without triggering Compose's measure / layout / composition phases. Visual effects like rounded corners and shadows can completely bypass recomposition during scrolling.

How to Verify Recomposition Behavior

Android Studio provides real-time tools to visually see which components are recomposing.

01

Open Running Devices + Layout Inspector

Android Studio → View → Tool Windows → Running Devices, connect a device and then open Layout Inspector.

02

Enable Recomposition Counts

At the top of Layout Inspector → enable "Show Recomposition Counts". Components being recomposed will be highlighted and counts update in real time.

03

Observe During List Scrolling

While scrolling, only newly visible items on screen should show recomposition counts. Off-screen items and items whose data has not changed should not increment their counts, confirming that skip optimization is working.

This Is Not Theoretical

You can open the tools and verify at any time. If a component shows an abnormally high recomposition count, it indicates a parameter stability issue -- check the @Stable annotations on its parameter types.
