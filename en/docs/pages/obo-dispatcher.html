OBO Dispatcher

The Queue Never Has More Than
One Task

Eliminates jank caused by message queue congestion while making each task an independent message, enabling performance monitoring to pinpoint "this SDK took 51ms."

Message Congestion Elimination Third-Party SDK Interception Source-Level Diagnostics ASM Bytecode Rewriting

Problem Background

Problem 1: Message Congestion → Jank
A large number of async tasks are submitted to the main thread queue simultaneously, flooding it instantly. Android sync barriers cannot interrupt already-queued tasks, rendering frames get blocked behind the queue, and frame drops become inevitable.
Problem 2: Jank Without Knowing the Cause
With dozens of tasks executing together, performance monitoring can only see "this batch took 200ms total" and cannot identify which specific task or SDK caused the slowdown.

Solution

The core principle of OBO (One By One): there is never more than one task in the queue. The next task is only submitted after the current one finishes.

Result

When VSYNC arrives, the queue is nearly empty, sync barriers work as intended, and rendering frames execute smoothly. With the same workload (30 list items x 10 tasks x 3ms), native concurrent dispatch causes severe jank, while OBO sequential execution maintains silky-smooth scrolling.

Trade-off

OBO increases the total completion time for a batch of tasks, but users won't notice background tasks finishing in "2 seconds vs 1 second" -- they will definitely notice "scrolling froze for 1 second." Smoothness over speed is the right trade-off.

Built-in Comparison UI

The stress test page includes a real-time frame rate indicator with adjustable parameters. Toggle between OBO and native mode with one tap -- the difference is immediately visible.

Handler Interception

OBO can only govern the project's own code. Third-party SDKs (IM / Payment / Attribution, etc.) call Handler.post/send directly to post messages to the main thread, bypassing OBO and still causing congestion.

Solution: compile-time ASM bytecode rewriting automatically redirects third-party SDK Handler calls through OBO, with no SDK cooperation required.

Compile time: INVOKEVIRTUAL Handler.post(Runnable) → LDC "com.thirdparty.SomeClass" ← caller class name injected at compile time → INVOKESTATIC OBOCompat.post(Handler, Runnable, String) Runtime: Immediate → OBOScheduler.post(tag) { r.run() } Delayed → handler.postDelayed({ OBOScheduler.post(tag) { r.run() } }, delay) Timed → handler.postAtTime({ OBOScheduler.post(tag) { r.run() } }, time) Front → OBOScheduler.postFirst(tag) { r.run() }

Core principle: timing is handled by the system MessageQueue; execution is handled by OBO.

Adding New Third-Party SDKs

They are intercepted by default. To exclude one, simply add its package prefix to OBOHandlerConstants.SKIP_PREFIXES.

Safety Guarantees

All classes in the com.example.archshowcase.core.scheduler.* package (including OBOCompat) are excluded from rewriting to prevent recursion. Non-main-thread Handler calls are passed through directly, and the timing semantics of delayed/timed messages are fully preserved.
